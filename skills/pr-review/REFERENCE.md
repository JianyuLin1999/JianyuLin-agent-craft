## §1 那 5 条核心理念(DNA)—— 看改动有没有跟它们打架

这 5 条是项目里最容易**浅看 PR 时你自己察觉不到**的硬规矩。每一条违反的形态都是"代码里没东西" —— 不是某行写错了,而是该有的东西没出现。所以你光顺着 diff 一行行扫永远扫不出来。

这恰好是 pr-review 这个 skill 真正高价值的地方:把这种"没出现的问题"挖出来。

### 1. A4 隐式知识断裂 —— "没写下来等于不存在"

**什么时候要警惕这一条**:PR 用了某条约定但仓库里查不到这条约定写在哪 / 假设某个本地状态存在但其实只在某个人的电脑上有 / 只有在场的协作者才理解的 context。

**真实例子**:PR #149。Yennfe 在自己电脑上看不见用户本地 `CLAUDE.md`(那个文件在 `.gitignore` 里,git 不追踪它),她以为这个文件不存在,就**自己重新写了一份 144 行的版本**。你审 PR 时光看 diff 完全看不出这件事 —— 因为 diff 里就是凭空多了 144 行,你不知道这是"重写已存在的东西",会以为是合理新增。

### 2. A10 + A40 静默路径 —— 选了路但不留痕,出错了不出声

**什么时候要警惕**:出错被 catch 之后只打个日志就继续走(没向上抛 / 没发事件 / 没记 metric);用户主路径上有选择分叉但没强制记"为什么选这个"在哪;关键失败状态没人订阅没人响应。

**真实例子**:#192 的渲染器外壳里,"问每个渲染器能不能处理这个文件"这个调用如果出错,catch 块直接吞掉异常静默继续,旁边注释还写着"暂不引入日志依赖"。这就是"静默吞错"的具体形态。你浅看 diff 不会察觉 —— 因为违反的形态就是"代码里没东西在响应这个失败"。

### 3. 0.1 + A39 硬编码 vs LLM 推理的错位

**什么时候要警惕**:PR 里冒出 N 步 / M 条 / K 个风险点这种硬阈值;或者反过来,本来该硬编码住的事却让 LLM 临场推理。

**真实例子**:翻译 prompt 里写"50 字以内"这种硬数字。`ls` 命令翻译过来 8 字就够了,复杂文档可能 50 字根本截不全 —— 同一个数字不可能同时合身。

### 4. A14 信任梯度错位 —— 当前选的档跟应该选的档不匹配

**什么时候要警惕**:PR 实现某个能力时选了"让 LLM 临时生成",但其实有更可靠的路径(做成 Skill / 预编译的资产 / 构建时就消灭掉)。

**真实例子**:翻译降级走"LLM 临时生成",但其实可以做成 Skill 提前验证好。同样的能力,前者不可控,后者可控,选错了档。

### 5. 0.2 + A22 活文档过时 —— 文档跟事实脱节

**什么时候要警惕**:PR 改了行为但 PRD / 代码注释 / handoff 文档没跟着改;或者反过来,文档里声明的"未来工作"已经没人做了但还挂在那里。

**真实例子**:`.gitignore` 的注释写着"Captain-level CLAUDE.md should not be in git",但实际上那份文件其实是 Dev 操作手册,不是 Captain prompt —— 注释跟事实早就脱钩了。

## §2 涟漪的 5 个方向 —— 想想这个改动会牵连到别的地方

每条 PR 都跑 5 个方向的涟漪嗅探。你拿到 `fetch-pr-context.sh` 打包好的 context 之后,**靠语义判断每个方向有没有真涟漪**(脚本不替你判断)。

### A. 往后影响 —— 这条 PR 改了 X,main 上以前依赖 X 的代码会怎样?

**怎么嗅**:diff 改了 API / 函数签名 / 数据结构 → 用 `Grep` 工具找 main 上还有谁在调用 / 依赖。

**真实例子**:#192 在 `FileBrowser.tsx:249` 把 `fileSize: 0` 当占位塞进去。Phase 4 的 json-tree 渲染器是按 fileSize 来分派的(小文件走纯文本,大文件走树状) —— 这行不改,Phase 4 启动后 json-tree **永远不会被命中**。

### B. 往前依赖 —— plan 后续 phase 还要动 X 吗?会不会冲突?

**怎么嗅**:对比 `plans/*.md` 里后续 phase 的范围,看跟当前 PR 改的文件 / 概念有没有交集。

**真实例子**:#192 引入了 `RendererHost`,Phase 2 docx 要在它基础上扩。`host.tsx` 顶部注释明示"DOMPurify shell 强制 改成 renderer 自调(v1 折中)" —— Phase 2 docx 必须记得调 sanitizeHtml,不然就漏过净化。

### C. 同类横向 —— 这条 PR 跟同时 open 的其他 PR 有交集吗?

**怎么嗅**:`fetch-pr-context.sh` 打包好的 context 里已经包含其他 open PR 的元数据 + 改了哪些文件。**你拿到文件交集后做语义判断**:是真涟漪(改同一段逻辑会撞车)还是假信号(改同一个文件但是完全不相关的两个函数)。

**真实例子**:#194 docx / #197 xlsx 都基于 #192 + 都在 `preview/` 文件夹下,合并顺序敏感(谁先合会影响 fixture 文件夹的 git history)。

### D. 配套文档 —— 这条 PR 改了代码 / 概念,显式声明文档跟上了吗?

**这一项你来做语义判断,不靠 grep 脚本**:

| 文档 | 你要看什么 |
|---|---|
| `handoff.md` | 测试基线 / 当前阶段 / 待办事项 改了吗 |
| `docs/changelog/` | 里程碑级 PR 漏写 changelog |
| `docs/prd/*` | PRD 写"v1 不做 X",PR 偷偷加了 X |
| `plans/*` | Plan 写"Phase 1 范围",PR 越界做了 Phase 2 的事 |
| `AGENTS.md` §7 文档地图 | 新建 PRD / changelog,§7 没登记 |
| 哲学母版 | PR description 里 cite 了 A22,但 diff 里没真正落地 |

**为什么这一项不能用 grep 脚本代劳**:文档不会用精确文件名描述代码。比如 PR 改了 `host.tsx`,文档里写的可能是"渲染器外壳" / "Error Boundary 组件" / "DOMPurify 净化层" —— **同一个概念被不同名字指代,grep 抓不到**。这一项必须你用语义识别。

### E. 历史经验 —— git history / 老注释 / 已废弃设计有没有被破坏?

**真实例子**:`.gitignore` 注释跟新文件用途脱节;函数 jsdoc 说"返回 null",实际抛异常;PRD 声明"放弃方向 Y",代码里还有 Y 的 dead branch 没清理。

## §3 工程债 —— 看着觉得哪里不够好就提一下

**这一节不给你穷举检查清单**。给你下面这段 prompt,让你自己判断:

```
任务:审查这条 PR 是否达到 ship 级质量。

判据:不只是"能跑",而是"作为长期维护的代码,是否真正足够好"。

哲学锚点(任何一条被违反就 surface,但你不限于这些):
- 单一权威:同一逻辑只在一个地方定义,其他引用不复制
- 抽象层级:没有过度抽象,也没有过度具体
- 隐性债显式化:stub / TODO / @ts-ignore / as any 都该有"为什么 + 何时改"注释
- 命名传达意图:避免 "util" / "helper" / "manager" / "data" 这种空名
- 控制流可读:嵌套深度合理,分支清晰
- 错误处理显式:不静默吞错
- 测试深度匹配代码深度

凡你判断"还可以更好"的地方,提出来。

你是质量守门员,不是温和提示者。**默认严苛** —— 提出来的事默认是 🟡 该改;
只有改进成本明显远大于收益(eg 改一行 vs 重构一周),才降 🟢。

对每条发现,问自己:"这是局部,还是模式?"
如果是模式,你提出来时应该呈现这层。

PR 里有没有"顺手改",你**高确信**认为帮了倒忙
(改错 / 引入混乱 / 破坏 scope 外的功能)?
只提这种。良性顺带(typo / 同任务相关的小整理)放过。
```

**这一节的核心精神**:开放判据 + 哲学锚点 + 你自由判断,我们不替你想检查清单。你拿到这段 prompt 自由发挥,**catch 我们都没想到的第 N 类问题**。如果我们给你列清单,你就只在那 N 项里转,看不到第 N+1 个问题。

### 3.1 机械再扫一遍这 7 个常见盲区(开放判据的兜底)

开放判据靠语义判断会出现"**阈值漂移**" — 你在"看上去都过得去"的时候自然就滑向"0 个发现"。这 7 维度是**一律机械扫**(用 grep / 模式匹配一律过一遍,不靠语义判断),专门 catch 开放判据会漏的常见小毛病:

| 维度 | 怎么扫 | 典型发现 |
|---|---|---|
| **① 类型安全** | grep `as unknown as` / `as any` / `// @ts-ignore` / `// @ts-expect-error` / `!` non-null assertion 在 prod 路径 | `as unknown as` 双 cast 跳类型系统;`!` 在用户输入路径会运行时炸 |
| **② 错误处理** | grep `catch.*{}` 空 catch / `catch.*console.` 只 log 不 rethrow / `try.*await.*}` 缺 catch / Promise 无 `.catch` | 静默吞错让用户看到默认值不知道失败了(违 A40) |
| **③ 无障碍配套** | grep `role=` / `aria-` 看 ARIA 是否配套(`role="tab"` 缺 `aria-controls` / `role="dialog"` 缺 `aria-labelledby` / `<img>` 缺 `alt`) | tabs `aria-selected` 但缺 `aria-controls` 指向 tabpanel |
| **④ 跨 codebase 一致性** | 改动里出现的 pattern(eg 用 `react-markdown` / `useCallback` / 错误模式),grep 项目别处怎么做这件事 | 别处 markdown 都传 `remarkPlugins=[remarkGfm]`,本 PR 没传 → 视觉不一致 |
| **⑤ 性能** | grep 改动里:递归遍历 / 嵌套循环 / 同步阻塞 / 大数组 `.filter().map().reduce()` 链 / `useCallback` 依赖数组里高频变化的对象 | `findFileSizeById` 每点击 O(n) 遍历;`useCallback` 加 `allData` 依赖每次重建 |
| **⑥ 文档同步** | diff 中改了:① 公开 API ② 行为契约 ③ 用户可见文案 ④ 配置项 → grep PRD / docstring / changelog / handoff.md 是否跟改 | description 里 "Phase 2 等 review",实际 Phase 2 已合 → 描述过时 |
| **⑦ 测试覆盖度** | 实现里的 discriminated union case(`type X = A \| B \| C`)/ enum / switch 分支,grep 测试里是否每个 case 都有 fixture | ipynb 实现支持 stream / execute_result / display_data / error 4 种 output type,测试只覆盖 stream |

**怎么用**:跑完 DNA / 涟漪之后,**按这 7 维度各跑一遍 grep / pattern match**,任何命中都进入候选池,再决定严重度。

**重要**:这 7 维度不是穷举,是**最常被 LLM 阈值漂移跳过的几类**。开放判据(§3 那段 prompt)仍然要跑,7 维度是兜底,不是替代。

### 3.2 如果你觉得这次完全挑不出毛病,先停下来反问自己三遍

如果你跑完 §3.1 的 7 个盲区 + 开放判据扫描后产出 0 个发现,**publish 前必须自答**:

1. **我有没有用"这个模式之前 phase 已经用过了"当借口跳过?**
   - 错的做法:"Phase 2 已经这么写了,所以 Phase 3 不再审这个模式"
   - 对的做法:每个 phase 独立审。前 phase 走过不等于本 phase 没新问题

2. **我有没有用"作者已经解释过为什么"当借口跳过?**
   - 错的做法:"作者 docstring 写了 lenient 解析是故意的,所以不提"
   - 对的做法:作者的设计选择仍然可以质疑。docstring 解释了 ≠ 设计就无可改进

3. **我有没有用"看上去合理就放过"当借口跳过?**
   - 错的做法:"没看到明显问题,放过"
   - 对的做法:"没看到"等于"我没扫到"。回去把 §3.1 7 维度全跑一遍

3 条都过 + 7 维度全扫过 = 这次确实挑不出 = **可以 publish(就写一份纯致谢评论)**。完美的 PR 是存在的 — **不要为了凑数硬找问题**,假发现比无发现更伤作者信任。否则,回去补发现,重新写草稿。

#### 一个复盘 example(#198 用新机制会怎样)

**旧 review 的产出**:0 发现,"教科书级落地"

**用新机制走完应该有的发现**:
- 🟡 **维度 ①**:`fakeBridge() { return { ready: true } as unknown as CaptainBridge }` 双 cast 跳类型;`getSkillSystem: async () => ({})` 同样的 stub 问题
- 🟢 **维度 ⑦**:测试只跑 happy path,handler 返回 404 / 路径遍历 / 权限错误这些错误路径没有回归保护(虽然本 PR scope 是 stream 不变量,但顺带提一下让作者知道这部分仍然是 gap)

→ 7 维度扫完真的找出了 2 个发现(原本以为是"教科书级"的 0 发现 PR 其实不是)。这就是反思自检的价值 — 不靠灵感,机械跑完才知道有没有漏。

## §4 PR 评论的文字怎么写 —— pr-review 特有的部分

通用可读性规则(白话 / 路径独立成行 / 编号禁入正文 / 完整段落对比)在 SKILL.md 顶部 inline 的 SHARED-READABILITY 里讲过了。这一节只讲 pr-review 特有的:评论结尾的备忘 / 严重度 icon / 自检判据。

### 在评论结尾给自己留一段备忘

这段备忘是**给你下次跑 review 时看的**,**作者看 PR 时 GitHub markdown 自动隐藏**(他完全看不到)。

```markdown
🟡 [白话标题]

[场景化解释,不堆术语]

位置:[人话描述路径]

—— 把代码跟设计的偏离主动写下来是好习惯。

<!-- audit: file=host.tsx, ripple=D-doc, dna=0.2, severity=yellow -->
```

备忘里这几个字段记什么:
- `file=<path>` —— 主要相关文件(让你下次一眼定位 + "只看新增"机制用)
- `ripple=<A|B|C|D|E>` —— 触发了哪个涟漪方向
- `dna=<原则编号>` —— 触发了哪条 DNA 原则
- `severity=<red|yellow|green>` —— 严重度
- `kind=<positive|concern>` —— 可选,标记这是正向观察还是问题点

### 严重度 icon

| icon | 含义 |
|---|---|
| 🔴 | **必须改,不改不能 merge** —— 违反 DNA / 严重隐性破坏(比如 stub 没标记,会让后续 phase 沉默踩坑) |
| 🟡 | **建议改** —— 默认大多数发现落这里。作者自己决定本 PR 改还是开新 PR 跟进 |
| 🟢 | **顺手提 / 正向观察** —— 真的很微小,或者改成本远大于收益;或者作者"做对了"的正向标识 |

**强制要求**:每个发现(不管是致谢正向 / 顺手提 / 必须改)**都必须挂一个 icon**。没挂的发现等于没分级,作者无法快速分流"哪些必须改 / 哪些可选"。心里挑剔层找出问题之后,嘴上说话层写出来时连同 icon 一起挂上 —— icon 是分流信号,不是装饰。

### 写完草稿之后的自检 —— pr-review 特有的判据

通用可读性自检在 SKILL.md inline 的 SHARED-READABILITY 里(reader 测试 + 7 条手段)。这一节只列 pr-review 特有的:

| 判据 | 容忍度 |
|---|---|
| 三档严重度大多数是 🟢 | **不允许** —— 默认严苛,大多数应该是 🟡 |
| 评论结尾的备忘漏贴 | **0** —— 每条发现都要贴 |
| 身份混乱(出现"建宇你看" / "skill 嗅探到") | **0** |
| reviewer 内部元信息(建议动作 / 评论预览 / 发现统计的文字)漏到 PR 评论里 | **0** —— 这些只能在终端报告 |

**再跑 [SHARED-COMMENT-VOICE.md §7](../SHARED-COMMENT-VOICE.md) 的 8 条判据**,0 容忍。两套自检都过才能 publish。

## §5 3 种草稿分别是什么

### 5.1 PR 评论(用建宇口吻直接对作者说)

这是**回应当前 PR**用的。**身份必须自洽**:用 "我"(= 建宇,质量守门员)和 "你"(= PR 作者)的口吻,不能出现身份混乱(详见 SKILL.md §1.1 人称纪律)。

#### 评论里要有哪几部分

**必填**(任何 PR 评论都要):

- **整体评价** —— 白话一句话,作者一眼知道这次 review 整体感觉
  - 例:"Phase 1 这版比 Phase 0 又高一档,可以直接合 + 立刻启动 Phase 2"
- **具体识别(致谢核心)** —— 至少 1 处"做对了的事",PR 越好识别越多(3-5 处都不嫌多)
- **发现清单**(没发现就写"未发现",显式)

**可选**(按场景填):

| 场景 | 加什么 |
|---|---|
| Merge + 启动下一个 Phase | 启动信号段(具体到"物质条件就位清单") |
| 留 🟡 改进 | 改进要求段 |
| 阻塞 🔴 | 阻塞原因段(具体到"为什么必须先改") |

#### ⭐ 高质量致谢提示词(merge 必须致谢,空话致谢比不致谢更伤)

**核心目标**:让作者感受到"我做的努力被精确看见了"。

**每条"具体识别"必须包含 3 个要素**:

1. **这处做了什么** —— 用具体动作命名(eg "把 X 决策搬到 Y 自己声明"),不堆术语
2. **为什么这件事重要 / 难做** —— 原本可能怎样,作者却做对了什么不容易的事
3. **这处做对了带来的连锁好处** —— 对未来 phase / 团队 / 项目长期质量的影响

**对比示范**:

❌ **空话致谢(违反提示词,永远不能这样写)**:
> "辛苦了,这个 PR 整体质量很好,继续保持。"
> "做得很棒,契约测试写得很完整。"

→ 作者读完毫无"被看见"的感觉,跟"已读"差不多。这种致谢比不致谢更伤 —— 因为它说明 reviewer 没认真读,反而打击作者下次认真做的动机。

✅ **高质量致谢(符合提示词)**:
> ### 1. 把"该不该拉文件内容"这个决策搬到每个渲染器自己声明
>
> 旧主组件里硬编码一个"二进制扩展名清单",每加一种格式都得改主组件。
> 现在每个渲染器自己声明"我需不需要预取文件内容",主组件查这个声明决定。
> **从此加新格式不用改主组件**。这是产品哲学里"加格式 = 一个新文件 + 一行注册"的真正落地。

→ 命名了具体动作 + 解释了为什么重要 + 连锁好处。作者读完知道:**我做的这件事被精确看见了,而且对项目长期质量有意义**。

**致谢提示词清单**:

- ✅ 用具体动作命名("把 X 搬到 Y")—— 不要泛泛"做得好"
- ✅ 说出"为什么这件事不容易"或"原本可能怎样"—— 让作者感受到"难度被识别"
- ✅ 连接到未来 phase / 团队 / 项目长期质量 —— 让作者感受到"我的贡献有影响"
- ❌ 不要空话("做得好" / "很棒" / "辛苦了" 单独成段)—— 比不致谢更伤
- ❌ 不要把要提的事 wrap 上"瑕不掩瑜"伪装成正面 —— 致谢和待提的事必须分开写
- ❌ 不要堆砌形容词("非常优秀的设计" / "极佳的代码") —— 用事实和后果说话,不用形容词

**merge 必须致谢,例外极少**:

即便是没什么亮点的小 PR(改个 typo / 升级 dep),也至少识别一处具体的事 —— eg "升级的版本号选了 patch 而不是 minor,符合最小风险原则" / "fixture 覆盖了 3 类失败情况,这种边界 case 经常被忽略"。

**不允许 merge 不带致谢**。

**草稿放在哪**:`~/.pr-review-drafts/pr-{N}-comment.md`

### 5.2 Issue 草稿(最简形态)

**什么时候要生成**:PR review 中发现了"这条 PR 之外的独立工作"(新的 bug 类型 / 推迟的事项要跟踪 / 系统性问题)。

**形态**:title + body,你自己判断写多详细。不强制结构。**等 issue-triage skill 设计完再丰富**。

**草稿放在哪**:`~/.pr-review-drafts/pr-{N}-issue-{slug}.md`

第一行必须是 `# <title>`(publish 脚本会提取作为 issue title)。

### 5.3 文档 patch

**两种形态(你自动选,user 不感知)**:

| 场景 | 形态 |
|---|---|
| **PR 内文档没跟上代码**(handoff / changelog / PR 内 docs) | **GitHub Suggestion**(嵌在 PR review comment 的 markdown 块里) |
| **跨 PR 文档过时**(PRD / philosophy 矛盾,跟某条具体 PR 没绑定) | **独立 patch 文件** |

#### GitHub Suggestion 形态

嵌在 PR 评论的 markdown 块里:
````markdown
建议改成:
```suggestion
新内容
```
````

作者点 "Commit suggestion" 一键应用。

**草稿放在哪**:这种 patch 是 PR 评论的一部分,直接放在 `pr-{N}-comment.md` 里。

#### 独立 patch 文件形态

完整的 `.patch` 文件,建宇会在自己的小 PR 里 `git apply` + commit。

**草稿放在哪**:`~/.pr-review-drafts/pr-{N}-doc-patch-{slug}.patch`

## §6 "只看新增"机制(skill 自动判断,user 无感)

| 走哪个 | 触发条件 |
|---|---|
| **只看新增改动** | 找到上次评论结尾的备忘(`extract-audit-tails.sh` 输出非空) **+** 距上次跑 < 14 天 |
| **从头审一遍** | 没备忘 **或** 距上次 ≥ 14 天 |

**关键原则**:

- **不记 user 的判断** —— user 弃掉的发现下次还会出现(user 可能改主意重新判)。这是设计意图,不是 bug。
- **只记 skill 自己跑过什么** —— 知道上次提了哪些事 / 作者改了哪些文件
- **正向信号** —— 你能提"上次提的 X 作者已经修复",作为正向反馈(用 🟢 kind=positive)

**14 天阈值**:超过 14 天 PR 大概率有重大变化,从头审一遍比"只看新增"更稳。

**escape hatch**(不显式公告):user 想强制从头审?手动从 PR 删掉评论结尾的备忘 comment,下次你就会从头审。

## §7 终端报告长什么样

### 终端报告的核心设计原则

**user 在终端必须一眼看到这 4 件事**(不需要 cat 草稿才知道):

1. **一句话整体判断** —— 这个 PR 整体什么情况
2. **建议你做的事** —— 明确的下一步动作 + 理由
3. **评论开头预览** —— 你准备替他发什么(开头 5-10 行)
4. **发现统计** —— 数量级感知

为什么这样设计:让 user 在终端 30 秒就能决定"是直接 publish 还是先 cat 完整草稿微调"。

### `/pr-review #N` 模式的终端报告

```
═══ PR Review Report ═══

📌 PR #194 / 作者 Yennfe
   feat(file-preview): Phase 2 docx 渲染器(截图痛点闭环)

💬 一句话判断
   docx 渲染做得很完整,9 文件 +364 行干净加法,baseline 全过,可以直接合。

🎯 建议你做的事:✅ 直接 merge + 留致谢评论
   理由:核心功能完整 / 测试覆盖 / 0 阻塞 / Phase 3 启动信号就位

📝 准备替你发的 PR 评论(开头预览):
─────────────────────────────────────────
✨ docx 渲染这版做得很完整,可以直接合。

具体识别(3 处做得真漂亮):

1. mammoth 用"用户打开 docx 才加载",首次启动包大小没受影响
2. 失败 fixture 三类齐全(损坏 / 加密 / 超大),边界 case 覆盖到了
3. ...
─────────────────────────────────────────
   完整草稿:cat ~/.pr-review-drafts/pr-194-comment.md
   一键发布:bash scripts/publish.sh #194

📊 发现统计
   🔴 阻塞:0   🟡 应改:1   🟢 观察:3(含 2 个正向)

📂 已生成草稿:
   - pr-194-comment.md           (PR 评论:致谢 + 1 🟡 + 3 🟢)
   - pr-194-issue-phase-3-prep.md (Phase 3 启动前的 1 件准备工作)
```

### 5 种建议动作(选 1 个,不要发明新的)

**注意:merge 永远配致谢评论,不存在"无需评论"档**(详见 §5.1 高质量致谢提示词)。

| icon | 动作 | 触发条件 |
|---|---|---|
| ✅ | **Merge + 致谢评论(必有致谢)** | 0 🔴 阻塞;🟡 应改事项可以延后到 follow-up issue / 后续 PR |
| 🟡 | **评论要求作者完善后再 merge** | 有 🟡 应改事项必须在本 PR 内改完(eg 启动信号缺一环 / 关键 stub 没标注) |
| 🔴 | **阻塞,等作者改完重审** | 有 🔴 阻塞事项(违反 DNA / 严重隐性破坏 / 回归测试破) |
| ⏸ | **暂缓拍板,需要 user 自己判断** | 设计层面的事你拍不准(架构选型 / 战略方向 / 哲学优先级) |
| ❌ | **建议关闭这条 PR** | 方向错了 / 重复别的 PR / 已经被新设计取代 |

**"建议动作"必须给具体理由**,不是空判定。例:

- ✅ 不写"approve",写"Merge + 致谢评论,理由:0 阻塞 + Phase 3 启动信号就位 + 契约测试覆盖完整"
- ❌ 不写"需要修改",写"评论要求作者补 mammoth 打包命名后再合,理由:不补的话 Phase 4 排查会绕路"

### 评论预览部分

预览**评论开头 5-10 行**,够 user 感知"语气是不是对路 / 重点抓得对不对"。

预览**必须包含**:
- 一句话整体评价(白话,不是术语)
- 至少 1 个"具体识别"或者发现标题(让 user 知道你准备说哪些事)

预览**不需要**完整发现列表(那是草稿文件的事)。

### `/inbox` 模式的终端报告

```
═══ Inbox Report ═══

✅ 建议直接合 (2 个 PR):
   #198 /api/raw 100MB 流式测试 — 0 发现,致谢即可
   #199 Windows CI runner — 0 发现,致谢即可

🟡 需要你拍板 (3 个 — 已深度审 + 生成草稿):
   #194 Phase 2 docx        — 1 🟡 / 2 🟢 — 建议直接 merge + 致谢
   #197 Phase 3 xlsx + ipynb — 2 🟡 / 1 🟢 — 建议要求补一项再 merge
   #189 prompt 修正         — 1 🔴       — 建议阻塞,等作者改完重审

⏳ 待 CI / 等作者 (4 个):
   #111 / #136 / #178 / #191

📂 已生成 6 份草稿,bash scripts/publish.sh #N 一键发
```

**跟 `/pr-review #N` 模式的关键差异**:inbox 是横向盘点,每个 PR 给一句话 + 建议动作即可,不嵌完整预览(预览在草稿文件里)。

### 草稿文件位置

`~/.pr-review-drafts/`:

```
pr-{N}-comment.md           # PR 评论(必有,即使是 "未发现 surface")
pr-{N}-issue-{slug}.md      # Issue(发现新独立工作时才有)
pr-{N}-doc-patch-{slug}.patch  # 独立文档 patch(跨 PR 文档过时才有)
```

## 反模式 —— 你设计 / 维护这个 skill 时不能做的事

**这一节是元层面提醒,你实施 / 维护 skill 时回看**。

### ❌ 不能把"依赖语境才能判断"的事硬编码进脚本

**好脚本** = 做一律照办的事(API 调用 / 字符串解析 / 集合运算)。

**坏脚本** = 做语义判断(比如 grep 文档涟漪、判断变量名堆积、判断哪些技术词需要翻译)—— 这些都依赖具体 PR 的语境,留给你来判断。

**反例**:`detect-doc-ripples.sh` 用 grep 找 PR 改的文件在 docs 里被提到的位置 —— 文档不会用精确文件名描述代码,grep 抓不到同义概念,**漏掉真实涟漪**。已删除,改让你用 `Grep` / `Read` 工具自己语义搜。

### ❌ 不能给你穷举检查清单替代开放判据

**工程债嗅探**绝不列"检查项 1-4"那种清单。给你哲学锚点 + "你不限于这些"判据,让你自由发挥。

清单化的代价 = 你在 N 个维度里转,看不到第 N+1 类问题。

### ❌ 不能让 skill 记忆 user 的判断 / 替 user 压制发现

**假阳性的解药是改 skill,不是压制发现**。

如果 user 反复弃掉同样类型的发现,意味着 skill 的某条判据该改。**改判据,不是给 skill 加"记忆"**。

### ❌ 不能写空话致谢(merge 必须致谢且有质量)

**空话致谢比不致谢更伤** —— 它说明 reviewer 没认真读,反而打击作者下次认真做的动机。

**禁忌**:
- "辛苦了,做得很好" / "继续保持" / "整体质量很高" 单独成段
- 用形容词堆砌("非常优秀" / "极佳的代码")
- 把 🟡 缺点 wrap 上"瑕不掩瑜"伪装成正面致谢
- merge 不带致谢

**正确做法**:每条具体识别走 §5.1 ⭐ 3 要素结构(做了什么 / 为什么不容易 / 连锁好处)。即便小 PR(改 typo / 升 dep)也至少识别一处具体的事。

### ❌ 设计 prompt 时默认倾向穷举(元层面留痕)

skill 设计 / 迭代时,**默认倾向会是"列出 LLM 可能要做的所有事"** —— 这恰好违反 A18 薄协议厚智能。

实证教训:本 skill grill 期间 reviewer 反复 catch 同一错误:

- 工程债嗅探最初列了 4 项检查 → reviewer 校准为"开放判据 + 哲学锚点"
- 假阳性收敛最初给 3 个选项铺满 → reviewer 校准 framing 本身错
- 反模式归纳最初给完整 prompt 模板 + 输出格式 → reviewer 校准"只做抽象提示词引导"
- 输入侧结构化最初推方案 B 多文件抽屉 → reviewer 校准让 LLM 做分类决策违反 A37

**判据**:写完任何 prompt 加法后,回看 ——
"我能不能用'你自由判断'四个字代替这 N 行?"

- 能 → 这 N 行是穷举陷阱,改简
- 不能 → 这 N 行是真锚点不是清单(eg 致谢 3 要素是真锚点 — 没有它,你写的致谢会变成"非常优秀的设计,继续保持"这种空话。3 个要素逼你指向具体的代码 / 文档 / 决策)

**列得越具体,你在那 N 个维度里转就越深,看不到第 N+1 类问题**。
