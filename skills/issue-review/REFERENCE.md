# issue-review 详细规则

SKILL.md 是简明的工作流,这里是细节 — 你做评估的时候随时翻回来查。

整套设计跟 pr-review / issue-create 是一家人:讲究 clarity-first / 零脑补 / 建宇口吻 / 默认挑剔 / 不写死封闭判据。

---

## §0 几条根上的原则,你时刻记着

### 0.1 你评估的对象是"想法和文字",不是"代码改动"

Issue 不是一段写好的代码,它是一个还在成形过程中的想法 — 可能还没想透,可能没说清楚,可能跟现有方向冲突而作者自己没意识到。所以你评估的角度跟 pr-review 不一样,pr-review 看代码改动的"涟漪",你看的是"内容本身够不够成熟"— 说清楚没、字段完整没、有没有可执行的下一步。

### 0.2 宁可多花 token,也别牺牲精度

拿背景资料的时候,你看到的脚本是把目标 issue 全文、所有 open issue、项目方向相关的全文文档全都给你拉过来 — 不是没省 token,是故意不省。issue-review 的价值正在于"看全了再判断",省下来那点 token 比起 review 出错的代价微不足道。

### 0.3 零脑补 — 你只能引用 issue 真的说过的话

你写出来的评论草稿里,只能引用 issue 实际写了什么、项目实际的文档里有什么。**不要凭印象编**"通用建议" / "标准做法" / "一般来说应该..." 这种套话 — 这些话对作者没用,而且暴露你没真读 issue。

### 0.4 建宇口吻 — 身份要自洽,不能穿帮

评论草稿是建宇的口吻、第一人称跟作者说话。**绝对不能在评论正文里冒出"skill 评估到""我作为 reviewer 检测到"之类的话** — 一冒出来作者就知道是 LLM 写的,信任感立刻崩。

### 0.5 评论说话的语气纪律(单源共享,pr-review 和 issue-review 共用)

写草稿的时候请遵循 [`~/.claude/skills/SHARED-COMMENT-VOICE.md`](../SHARED-COMMENT-VOICE.md) — 那里讲了 3 条核心原则、两个对立失败模式、致谢要能指着具体东西夸、别替对方想、字数和密度怎么把握、8 条自检。**这一份是 pr-review 和 issue-review 共用的,在那边维护,这边引用就好**。

里面最关键的几条洞察(也是最常错的):

- **说事实,别评价人**:正确的中间地带是只说事情、不评价对方
- **给选择,别下命令**:把决定权还给对方
- **不替对方想**:不预测对方反应 / 不替对方找借口 / 不替对方切活 / 不替对方估工作量

## §1 从这 5 个角度评估

每条 issue,你都按这 5 个角度过一遍。**关键:你靠语义理解去判断,不是靠脚本 grep 关键词**。原因很简单 — 同义概念 grep 抓不到。"渲染"和"预览"在 codebase 里其实是同一件事,但 grep 会把它们当两件事;只有你用语义读懂上下文,才能正确判断。

### A. Clarity — 说人话吗?

**怎么判**:想象作者本人 3 个月后自己回看,或者一个新进项目的协作者第一次读到这条 issue,他们站在地铁里,手机屏幕一巴掌大,每段 30 秒能不能搞明白"这是什么事 / 为什么提 / 现在该做啥"。这就是 clarity 的底线。

**这种情况算挂了**:

- 标题或正文堆满英文技术词 / 文件路径 / 变量名,看不到一句白话
- 那个"人话摘要"字段实际写的是 git commit message 风格("feat: add X")— 这不是给人读的,是给机器读的
- 推理链跳步,中间步骤丢了 — 比如作者写"因为 A 所以 D",但 B 和 C 在哪去了?读者得自己脑补

**这种情况算过了**:

- 标题就是大白话,例如 `[PRD] ipynb 预览应该按 cell tag 过滤隐藏调试内容` — 一眼能看出在说啥
- 正文里有具体场景描述,比如"我作为研究用户打开一个 100+ cell 的 notebook,看到所有 cell 平铺..."— reader 能跟着想象出画面
- 文件路径独立写成"位置:"那一行,不混在白话里把句子搅乱

### B. 完整度 — Sedna template 该填的字段填了吗?

**怎么判**:Sedna 有 3 个 issue 模板 — bug-report / prd-proposal / project-gap。每个模板都有几个必填字段。你对照这条 issue 用的模板,逐个 check 必填字段有没有真的填上了。

**这种情况算挂了**:

- 字段值还是 `🚧 待 grill 补完`(从 `/issue-capture` 出来的占位 issue 经常这样)
- 必填字段直接空着(template 要求填但作者没填)
- 必填字段"填了字但等于没填"— 比如 "TBD" / "暂无" / "稍后补",这种敷衍跟空着没区别

**这种情况算过了**:

- 所有必填字段实际写了作者自己的话,有具体内容
- grill-me 存档完整(prd 和 gap 模板强制要求附 grill-me 对话记录)
- bug 模板里有 bundle + 截图 + 完整重现步骤

### C. 唯一性 — 跟现有 issue 撞不撞?

**怎么判**:你扫一遍所有 open issue + 最近 90 天关掉的 issue,**靠语义判断**这条新 issue 是不是在重复别人已经提过的事。再说一次,不靠 grep — "渲染"和"预览"是同一件事,grep 抓不到,你能。

**这种情况算挂了**:

- 跟某个还 open 的 issue 完全是同一件事(应该合并到那条,不是另开一条)
- 跟某个已经关闭的 issue 是同一件事(已经被解决了,应该 reopen 那条或关闭当前条)
- 跟某个 placeholder 重复(应该把那个 placeholder 升级,不是新开一个)

**这种情况算过了**:

- 这条 issue 的主题在 open 和 closed 里都找不到相似对象,是个新东西
- 跟某些 issue 有关联但**不是重复** — 比如是子 issue / 父 issue / 横向相关,这种关联应该在评论里点出来,但不算"撞"

**注意一个跟 issue-create 的区别**:

- issue-create 的"重复检测"是 **事前** 的,目的是避免重复创建一条新 issue
- issue-review 的唯一性这个角度是 **事后** 的,目的是 catch 已经被创建但其实是重复的 issue

### D. 可执行性 — 读完知道下一步该做什么吗?

**怎么判**:任何 reader(包括 3 个月后的自己)读完这条 issue,**能不能说出来"接下来该做什么 / 怎么算完成"**。如果读完一脸懵,这个角度就挂了。

**这种情况算挂了**:

- gap 模板的"建议下一步"字段没填
- prd 模板的验收标准模糊,要求 3-7 条具体可检查的条目但作者只写了一句话
- 整条 issue 只到"我看到一个问题"为止,没说"我建议..."、没说"怎么算解决"

**这种情况算过了**:

- 验收标准用 `[ ]` checkbox 一条条列出来,每条都是可以打勾验证的具体条件
- bug 模板里明确写了"期望行为"对照"实际行为",reader 一看就懂哪里出问题
- prd 模板的"什么算完成"那栏列了 3-7 条具体判据

### E. 跟项目方向一致 — 跟 PRD / plan / 哲学对齐吗?

**怎么判**:背景资料里包含项目的 PRD、plan、哲学母版。你拿到这些之后,用语义判断这条 issue 跟项目方向是不是一致 — 是顺着方向走的,还是悄悄走反方向的。

**这种情况算挂了**:

- 跟某条 PRD 里"v1 不做 X"的显式声明冲突
- 跟当前 plan 的 phase 范围越界(eg phase 3 还没动,issue 提的是 phase 5 的事 — 不一定挂,但至少要提出来)
- 重新发明一个已经存在的功能(说明作者没看到已有方案)
- 跟某条哲学原则冲突 — 比如 PRD 写"避免 LLM 临场推理",但这条 issue 提议"让 LLM 临场生成 X"

**这种情况算过了**:

- issue 里引用了相关 PRD 或 plan(eg "对应 docs/prd/file-preview.md §3.2") — 看得出作者读过项目方向
- 在当前 phase 边界内
- 跟项目哲学一致(eg "符合哲学里'活文档'的精神,代码改了文档也跟着改")

### 5 个角度对所有 issue 都跑,但不同模板侧重不一样

你对所有 issue 都按 5 个角度过一遍,但**不同模板的重点不同**(由你自己根据 issue 内容判断,不写死规则):

- **bug 类**:B 完整度(有没有 bundle / 重现步骤)+ D 可执行性(期望行为是否明确)是重头戏
- **prd 类**:A clarity + B 完整度(有没有交代代价 / 备选)+ E 项目方向是重头戏
- **gap 类**:A clarity + D 可执行性(有没有"建议下一步")是重头戏

### 每个角度的结果,用三个 icon 之一

| icon | 含义 |
|---|---|
| ✅ | 这个角度过了 |
| 🟡 | 这个角度有问题,应该改 |
| 🔴 | 这个角度严重出问题,阻塞推进 |

跟 pr-review 一样,**默认挑剔** — 千万别为了让 issue 过审而把 🟡 偷偷降成 ✅。质量守门员的价值就在这,守不住就没意义了。

### 1.6 机械再扫一遍这 7 个常见盲区(不靠灵感,一律扫)

5 个角度的评估靠你做语义判断,但是 — 人(以及 LLM)有一个老毛病:**标准会偷偷漂移**。看了一阵子 issue 之后,你的"还行"基线会无意识地下移,本来该挑出来的 nit 就跳过了。

这 7 个盲区是机械扫的兜底 — 跑完 5 个角度评估之后,你必须再按这 7 项一项一项过,不是看灵感,是一律扫。哪一项命中了就提出来,挂上严重度。

| 盲区 | 你怎么扫 | 典型会发现的点 |
|---|---|---|
| **① 描述够不够精确** | 找模糊词:"等" / "之类" / "可能" / "类似" / "比较 X" / 没量纲的形容词(快 / 慢 / 大 / 小);看作者是不是说了"做某某功能"但没说具体边界在哪 | "改进 captain 表现" — 没说改什么 / 怎么算改进了 |
| **② 跟现有 open / closed 重不重** | 拿 issue 标题 + 关键词,语义比对 open issues + 最近 90 天 closed + 现在 open 的 PR 标题。任何概念重叠到一定程度就提出来 | "支持 xlsx 预览" 跟 closed #155 重复 |
| **③ Sedna template 必填字段** | 看这条 issue 用的 template(bug / prd / gap)有哪些必填字段,挨个看是不是 🚧 占位 / 空着 / 一句话敷衍 | bug 模板"重现步骤"只写"打开就崩" → 信息严重不足 |
| **④ 怎么算完成 / 验证标准** | 找"如何验证" / "怎么算完成" / "成功标准"这一栏;找不到就提出来 | PRD 没写"做完后怎么判断它有效" — 这种 issue 永远做不完 |
| **⑤ 跟项目方向有没有冲突** | 拿 issue 标题关键词去比对 `docs/prd/` 下的现有 PRD、`docs/philosophy/` 下的哲学原则,看有没有跟项目反方向走 | "用户手动管理 mamba 环境" 跟 A27 "障碍消除默认" + Q5 pivot "Agent autonomy default" 冲突 |
| **⑥ Scope 大小** | 看 issue 描述多长 / 涉及多少模块 / 跨多少 phase。涉及 > 3 个模块或 > 1 个 sprint = 该拆;< 5 行描述且只动单个文件 = 可能该合并到现有 issue | PRD 横跨 6 个模块 → 应该拆成 6 个子 issue |
| **⑦ 实物证据够不够** | bug 类看有没有 bundle / repro / 截图 / log;PRD 类看有没有 design doc / mockup / RFC link;gap 类看有没有附"我看到了什么现象" | bug 只有文字描述没附 log → 协作者复现得靠访谈作者 |

**怎么用**:跑完 5 个角度的评估之后,**按这 7 个盲区一项一项扫**,任何命中就放进发现的候选池,挂上严重度。

### 1.7 如果你觉得这次完全挑不出毛病,先停下来反问自己三遍

如果你跑完 5 个角度 + §1.6 的 7 个盲区后真的 0 个发现,**别急着 approve,先问自己这 3 个问题,坦诚回答**:

1. **我是不是用"作者写得很完整"当借口跳过了?**
   - 错的做法:"template 字段都填了,看上去都好" → 直接 approve
   - 对的做法:字段填满 ≠ 内容到位。完整度只是 1 个角度,verify 一下另外 4 个角度 + 7 个盲区是不是真的都过了

2. **我是不是用"主题听上去合理"当借口跳过了?**
   - 错的做法:"PRD 写得很专业,方向也对" → 直接 approve
   - 对的做法:方向合理 ≠ 这个 issue 本身充分。issue 是 issue,方向是方向,两件事

3. **我是不是用"5 个角度都过了"当借口跳过了?**
   - 错的做法:"5 个角度评估全过,没什么可挑的" → 直接 approve
   - 对的做法:5 个角度只是底线,不是天花板。回去把 §1.6 那 7 个盲区也机械扫一遍

3 个反问都过 + 5 角度 + 7 盲区全扫过 = 这次确实没发现 = **可以 ✅ Approve(就写一份纯致谢评论)**。完美的 issue 是存在的 — **不要为了凑数硬找问题**,假发现比无发现更伤作者信任。否则,回去补发现,重写草稿。

## §2 5 种建议动作 + 致谢怎么写

### 2.1 5 种建议动作(跟 SKILL.md 里那张表一致)

| icon | 动作 | 什么时候挑这个 |
|---|---|---|
| ✅ | Approve & 致谢评论(等开 PR 推进) | 5 个角度全过(或只有几条 🟢 顺手提) |
| 🟡 | 评论要求作者完善 | 有某个角度 🟡 |
| 🔧 | 建议升级 / 拆分 / 合并 | 是个 placeholder 该升级 / 太大该拆 / 重复该合 |
| ⏸ | 暂缓,需要建宇拍板 | 设计层面的方向选型 / 战略 / 哲学优先级 |
| ❌ | 建议关闭 | 重复 / 过时 / 错误方向 / 已经解决 |

### 2.2 ⭐ Approve 的时候,致谢怎么写才不流于客套(必读)

**你要达到的效果**:作者读完致谢之后,感觉"我的想法被精确看见了"。Approve 不配致谢 = 等于暗示你这个 reviewer 根本没认真读,**比不致谢还伤**(作者会觉得被敷衍)。

打个比方:你像一个老师,学生交了一份论文,你给了"通过"。如果你只写"通过",学生不知道你到底读没读;但如果你说"你这一段关于 X 的论证我特别欣赏,因为通常学生会跳过 Y 这一步,你却 catch 到了" — 学生立刻知道你不仅读了,还读到了关键的地方。这就是致谢的价值。

#### 致谢里每个"具体识别"必须包含 3 件事

1. **这条 issue 到底抓到了什么** — 用具体名词命名 issue 真正解决的问题,不要泛泛说"提了个好建议"
2. **为什么这件事 catch 到不容易** — 通常的视角会怎样,作者做对了什么
3. **catch 到这件事之后,会带来哪些连锁好处** — 对当前 phase / 团队节奏 / 项目长期质量有什么影响

#### 5 个常见的致谢角度,你自由选 1-3 个

| 角度 | 什么 issue 适合 | 例子 |
|---|---|---|
| **抓到关键 insight** | 触及到一个非显而易见的关键问题 | "你看到的这个问题其实是 file-preview 整体策略的一环" |
| **写得清晰** | issue 本身写作质量高 | "你这条 issue 一读就懂,'人话摘要'写得特别到位" |
| **完整度高** | template 字段全填了 + grill 存档完整 | "字段都填全了,grill 存档也在,后续 triage 节奏会快很多" |
| **timing 对** | 早期发现、还有时间消化 | "提前 catch 这个问题,Phase 4 启动前刚好有时间消化" |
| **主动反抗惰性** | 容易被压在心底的需求,作者主动提出 | "这种'好像可以但好像也无所谓'的需求容易被压在心底,你显式提出来值得肯定" |

#### 对比一下好坏写法

❌ **空话致谢(永远不要这样写)**:

> "好建议,赞同。"
> "这个问题确实值得关注。"
> "辛苦了,继续保持。"

作者读完没有任何"被看见"的感觉。这种致谢比根本不致谢还伤 — 因为它显得 reviewer 是在走形式。

✅ **高质量致谢**:

> 你这条 issue 抓到了一个研究用户的真实痛点 ——"想看分析主线但被调试 cell 干扰"。
> 这种需求在普通工程视角下容易被压在心底("毕竟可以滚动跳过"),但你显式提出来 + 给出"过滤 hidden + draft tag"的具体方向,后续 Phase 4 启动 ipynb 渲染时**直接有方案可落地,不需要再 grill 一轮**。

→ 你看 — 命名了具体痛点 + 解释了为什么这事容易被忽略 + 连锁好处。3 件事都有,读起来作者立刻知道你真的读了。

#### 千万别(跟 pr-review 共享的反模式)

- ❌ 空话致谢("好建议" / "辛苦了")
- ❌ 堆形容词("非常优秀的想法")
- ❌ 把改进的发现包装成"瑕不掩瑜" — 这是稀释你提的发现
- ❌ Approve 不带致谢
- ❌ 致谢和改进发现混在一起写,reader 分不清

## §3 issue-review 特有的几条格式规则

通用可读性规则(白话 / 文件路径独立成行 / 内部编号不泄露到正文)在 SKILL.md 上半截的 SHARED-READABILITY 段里。本节只讲 issue-review 独有的。

### 3.1 评论顶部用 emoji 起手,不要用 markdown H1

评论正文最上面那一行,用一个 emoji + 一句话整体判断起手 — 像这样:

> ✅ **这条 issue 整体方向清晰,信息也够,可以等开 PR 推进了。**

> 🔧 **这是个 placeholder,信息只填了 1/8,建议先跑 `/issue-create #220` 升级再 triage。**

为什么不用 markdown H1?因为 H1 在 GitHub issue 评论里渲染出来字号过大、看着像标题党;一行 emoji + 加粗一句话,信息密度刚好。

### 3.2 评论结尾给自己留一段备忘 — GitHub 自动隐藏

每个发现的最后,你贴一段 HTML 注释。这段注释 GitHub markdown 自动会隐藏掉,作者看不到,但你(或者建宇)将来回看这条评论时能 grep 出来 — 等于你给自己留的一段"我当时为什么这么挑"的备忘。

写法长这样:

```markdown
🟡 **某个角度发现的标题(白话)**

[场景化的解释]

位置:[白话描述]

—— [一句简短建议]

<!-- audit: dim=B-completeness, severity=yellow, ref=template-prd -->
```

里面 3 个字段:

- `dim=<A|B|C|D|E>` — 这条发现是从哪个角度触发的
- `severity=<red|yellow|green>` — 严重度
- `ref=<可选,你引用了哪条 PRD / plan / 哲学>` — 给将来追溯用

**这段注释是 HTML 注释,GitHub markdown 一律会隐藏**,作者只看到上面的白话内容,不会看到这些内部元信息。

## §4 Draft 文件长什么样

### 4.1 文件放在哪

`~/.issue-review-drafts/issue-review-{N}-comment.md`

### 4.2 文件格式

```markdown
---
target_issue: 220
suggested_action: approve | request_changes | restructure | hold | close
labels_to_add: []
labels_to_remove: []
close: false
generated_at: 2026-05-24T20:40:00Z
---

[评论正文 — emoji + 一句话整体判断 + 致谢 / 发现 / 下一步建议]
```

### 4.3 顶上 YAML 头部各字段的含义

| 字段 | 取值 | 用途 |
|---|---|---|
| `target_issue` | issue 编号 | 发布脚本校验跟命令行参数是否一致 |
| `suggested_action` | `approve` / `request_changes` / `restructure` / `hold` / `close` | 给发布脚本用 + 给建宇终端报告引用 |
| `labels_to_add` | `["a", "b"]` 或 `[]` | 发布时调 `gh issue edit --add-label` |
| `labels_to_remove` | 同上 | 发布时调 `--remove-label` |
| `close` | `true` / `false` | 发布时是否要 `gh issue close` |
| `generated_at` | ISO 时间戳 | 留痕,方便日后回看 |

这些字段顶上看似是"机器读的",但实际上是建宇做 review 时一眼扫的"标签" — 他看到 `suggested_action: restructure` 就知道你建议升级,看到 `close: true` 就知道你建议关 issue。所以字段值必须如实反映你的判断,不能含糊。

### 4.4 评论正文怎么组织

```markdown
[emoji 判定] **一句话整体判断**

[1-2 句承接 — 用建宇口吻,例如 "辛苦了,我从头看完整条 issue。整体方向..."]

## 具体识别(如果是 Approve)/ 该补的(如果是 🟡)/ 我的建议(如果是 🔧/⏸/❌)

[Approve 类:按 §2 致谢的 3 要素组织;其他类:按 §1 的 5 个角度组织你发现的问题]

[每条发现末尾贴上 §3.2 那段给自己留的备忘注释]

## 下一步建议

[一段白话,跟 suggested_action 对应的具体动作 — eg approve 就写"等你开 PR",restructure 就写"建议跑 /issue-create #N 升级"]
```

## §5 你在终端给建宇打印的报告长什么样

### 5.1 报告分 4 块(跟 pr-review 是一个套路)

```
═══ Issue Review Report ═══

📌 #220 [PRD] ipynb 渲染应该考虑 cell tag 过滤 / 作者 jianyulin / placeholder=true

💬 一句话判断
   占位 issue 内容方向清晰但 5/7 字段 🚧,该走升级流程而不是直接 approve。

🎯 建议你做的事:🔧 建议升级 (跑 /issue-create #220)
   理由:placeholder + 字段缺 / 5 维度只有 A clarity 过

📝 准备替你发的评论(开头预览):
─────────────────────────────────────────
🔧 **这是个 placeholder,字段大部分还是 🚧,建议先升级再 triage**

辛苦了。我看了一下这条 capture 出来的占位:方向清晰
("过滤 cell tag"),但具体哪些 tag / 怎么呈现给 user / 边界 case 都还没说...
─────────────────────────────────────────
   完整 draft:cat ~/.issue-review-drafts/issue-review-220-comment.md
   一键发布:bash scripts/publish-issue-review.sh 220

📊 5 个角度的评估
   A. clarity:✅
   B. 完整度:🟡 5/7 字段 🚧
   C. 唯一性:✅
   D. 可执行性:🟡 无验收标准
   E. 项目方向:✅(跟 PRD #155 file-preview matrix 对齐)
```

### 5.2 评论预览这一块的判据(一律按这个来,不靠灵感)

预览段落显示**前 5-10 行**就够了 — 建宇看了能感知"语气对路没 / 重点抓得对没"。

预览**必须包含**:

- 一行 emoji + 整体判定(白话)
- 至少 1 个"具体识别"或"该补的"或"我的建议"的标题

预览**不需要**列出所有发现 — 那是草稿文件的事,建宇真要看细节就 `cat` 那个文件。

## §自检 — issue-review 特有的几条判据

通用可读性的自检(reader 测试 + 7 条手段)在 SKILL.md 上半截的 SHARED-READABILITY 段里。本节只列 issue-review 特有的几条:

| 判据 | 容忍度 |
|---|---|
| **脑补内容**(你编了 issue 没说过的事) | **0** — 直接删掉,或者改成"我不确定..." |
| **第三人称泄露身份**(评论里冒出"issue 作者说"/ "skill 评估到") | **0** — 改回第一人称 |
| 评论结尾给自己留的备忘漏贴了 | **0** — 每条发现都要贴 |
| **建议动作不是 5 种之一** | **0** — 必须从 ✅🟡🔧⏸❌ 里选 |
| **Approve 不带致谢** | **0** — 必须有致谢 3 要素 |
| YAML 头部字段不完整 | **0** — target_issue / suggested_action / close 必填 |

**再跑一遍 [SHARED-COMMENT-VOICE.md §7](../SHARED-COMMENT-VOICE.md) 那 8 条判据**,0 容忍。两套自检都过了,才能让建宇 publish。

## §反模式 — 设计这个 skill 或者你执行任务的时候,千万别做的事

继承 pr-review / issue-create 的通用反模式 + issue-review 自己 3 条。

### 通用反模式(继承)

1. ❌ 不要把"看上下文才能判断"的事硬编码进脚本(5 个角度的评估、唯一性匹配 — 都让你做,不是脚本做)
2. ❌ 不要给你穷举一份检查清单,代替开放性判据(穷举会让你只盯清单,反而漏掉清单外的)
3. ❌ 不要让 skill 记忆建宇的判断 / 替建宇抑制某些发现(他每次决策都该是独立的)
4. ❌ 不要写空话致谢
5. ❌ 设计 prompt 的时候不要默认倾向穷举

### issue-review 特有的 3 条反模式

#### ❌ 不能用 grep 做唯一性匹配

C 角度(唯一性)**必须**你用语义来匹配,不能靠脚本 grep 关键词。原因前面讲过 — "渲染"和"预览"是同一件事但词不同,grep 抓不到,你能。

#### ❌ 不能跳过 E 角度(跟项目方向一致)

E 角度需要你读项目 PRD / plan / 哲学,token 成本最高,**但绝对不能跳**。Sedna 项目最核心的价值就是"跟方向对齐";一个跟方向冲突的 issue 被你 approve 进来,日后修复的代价远远大于 review 时多花的那点 token。这笔账要算清楚。

#### ❌ 不能替建宇做"是否真重复"的最终决策

C 角度发现疑似重复的时候 → 你提出来 + 给推荐,但**最终判决权给建宇**(他可能选 🔧 合并,也可能仍然 approve 当独立 issue 处理 — 看他对这两条 issue 的横向理解)。这跟 pr-review 那边"不让 Agent 做分类决策"的原则是一致的。

## 附:跟 pr-review / issue-create 是什么关系

| 维度 | pr-review | issue-create | issue-review |
|---|---|---|---|
| 评估对象 | 代码改动 | 想法草稿 | 已存在的 issue |

| 致谢对象 | 代码贡献(具体动作) | (无致谢概念) | 想法贡献(insight / clarity / timing 等) |
| 评论结尾给自己留的备忘 | ✓ | (无,创建不需要这个) | ✓ |
| 自检不可跳 | ✓ | ✓ | ✓ |
| 建宇口吻 | ✓ | ✓ | ✓ |
| 零脑补 | ✓ | ✓(更严苛) | ✓ |
