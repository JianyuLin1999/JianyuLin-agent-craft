# 审查严苛度 — 双层架构

reviewer(无论审 PR / issue / RFC / 设计文档)拿到一份产物时,**审查代码的态度** 和 **回复作者的态度** 是两件事。这份文件写第一件:**怎么对代码 / 想法 / 文字毫不留情**。

跟 [`comment-voice.md`](./comment-voice.md) 配套用 —— 一个管"找出多少",一个管"怎么说出来"。

---

## 核心一句话

**审查狠 ≠ 说话狠**。两件事**物理分离**,默认假设"还有改进空间没找到",挑剔上不封顶。

---

## §1 双层架构

| Layer | 对什么 | 态度 | 规则在哪 |
|---|---|---|---|
| **A 审查层** | 对**代码 / 想法 / 文字** | 默认假设"还有问题没找到",挑剔不封顶 | 本文件 |
| **B 措辞层** | 对**作者本人** | 说事实 / 带"我" / 给选择,温和不指控 | [`comment-voice.md`](./comment-voice.md) |

**关键洞察**:`comment-voice.md` 里的"指控 vs 委婉是对立失败模式"是 **Layer B 的纪律**(对人怎么说),**不是 Layer A 的妥协理由**(对代码可以放过)。两个 layer 同时做到。

### 典型分离示例

**Layer A 找到**:`fakeBridge()` 用 `as unknown as CaptainBridge` 双 cast 跳过类型系统,handler 哪天访问 `bridge.X` 会运行时炸

**Layer B 写出来**:

> 🟡 fakeBridge 用了 `as unknown as` 双 cast 跳过类型,handler 哪天 access `bridge.captainSession` 之类会运行时炸 undefined 而不是编译时拦。两条路你挑:A 改成 `Partial<CaptainBridge>` 让类型系统有 hint,B 暂时不动等真出问题再说。

审查毫不留情把问题挖出来 ✅,措辞说事实 + 给选择 ✅。

---

## §2 心理基线:至少 3 个可改进点

任何 PR / issue 在没找到至少 3 个 surface 之前**都不算审完**。

不是"凑数",是**强制扫描深度**。LLM 默认在找到 1-2 个就停手,这条逼它把维度全跑完 + 开放判据再扫一遍。

**例外**:极端简单的产物(改 1 个 typo / 删 1 个 dead import / 纯 dup-of close)可以 < 3。但必须先过下面的"0 nit 反思"。

---

## §3 产出 0 nit 时强制反思 3 条

如果跑完审查产出 0 nit,**publish 前必须自答**:

1. **我有没有用"已验证过的模式"借口跳过?**
   - 反例:"Phase 2 已经这么写了,所以 Phase 3 不再审"
   - 正确:每个 phase / PR / issue 独立审,前者过了不等于本次没新问题

2. **我有没有用"作者解释过为什么"借口跳过?**
   - 反例:"作者 docstring 写了 lenient 设计是故意的,所以不提"
   - 正确:作者的设计选择仍可质疑。解释 ≠ 无可改进

3. **我有没有用"看上去合理"借口跳过?**
   - 反例:"没看到明显问题,放过"
   - 正确:"没看到" = "我没扫到"。回去把维度全跑一遍

3 条都过 + 维度全扫过 = 真 0 nit,可以 publish。**否则补 surface 重新 draft**。

---

## §4 严重度三级强制挂级

每个 surface(不论致谢正向 / nit / 阻塞)**必须挂级**。无 icon 等于没分流,作者无法快速判断"哪些必改 / 哪些可选"。

| icon | 含义 | 典型场景 |
|---|---|---|
| 🔴 | **阻塞,必须改** | 违反核心原则 / 严重隐性破坏 / 回归 |
| 🟡 | **应该改** | 默认大多数 surface 落这里,作者决定本次改还是开新跟进 |
| 🟢 | **可选 / 正向观察** | 真微小,或改成本远大于收益;或"做对了"的正向标识 |

挂级是**分流信号**,不是装饰。Layer A 找出问题,Layer B 写出来时连同 icon 一起挂上。

---

## §5 维度扫描清单(deterministic 兜底)

开放判据靠语义判断会**阈值漂移** —— LLM 在"看上去都过"时自然滑向 0 nit。维度扫描是**机械 grep / pattern match**,catch 开放判据会漏的常见 nit。

**具体维度清单跟着 reviewer 场景走**(PR review 是 7 维 / issue review 是另一套 7 维 / 设计文档 review 又是另一套),不在本文件穷举 —— 在对应 skill 的 REFERENCE.md 里。

**通用要求**:跑完开放判据后,**按本场景的维度清单各跑一遍**,任何命中进 surface 候选池,挂严重度。

---

## §使用方式

两种方式,优先选 1:

**方式 1(推荐):整段复制 §1-§4 进 skill prompt**

agent 在 prompt 加载时拿到完整规则,不依赖跟链接 / 跟路径。集成成本 5 秒。

**方式 2:在 skill 里链接引用本文件**

适合仓库内运行的 skill。skill SKILL.md 加一句:"审查严苛度纪律见 `knowledge-assets/review-rigor.md`,默认假设还有改进空间没找到"。

### 跟 comment-voice.md 的配套

两份文件**必须一起读**。任何 reviewer skill prompt 都应该:
- 引用 `review-rigor.md`(审查层心态)
- 引用 `comment-voice.md`(措辞层纪律)
- 在 skill 内部把两层物理分离(不同章节,不同自检)
